# 프로젝트 폴더 셋업 규칙

## 원칙
1. **Monorepo + 역할별 폴더.** 목적이 분명한 곳에만 파일을 둔다.
2. **구조가 곧 품질이다.** 구조를 지키면 시간이 갈수록 AI 산출물이 좋아진다(구조 없으면 그 반대).
3. **좋은 구조는 점점 단순해진다.** 설정·폴더가 계속 복잡해지면 잘못 가고 있다는 신호.

## 표준 레이아웃

```
{{project}}/
├── src/        # 비즈니스 로직
├── docs/       # 사람이 읽는 문서 (비즈니스 진실·아키텍처·스펙)
├── tests/      # 검증 인프라
├── .dev/       # AI 작업 기록 (learnings·디버깅·스크래치)
├── .claude/    # AI 설정 (rules·skills·hooks·agents)
├── out/        # 빌드 산출물 (자동 생성, 수정 금지)
└── CLAUDE.md   # 프로젝트 지도 (≤120줄, docs/·rules/로 포인팅)
```

## 어디에 무엇을 두는가 (배치 규칙)

| 만드는 것 | 위치 |
|-----------|------|
| 비즈니스 로직 | `src/{도메인}/...` |
| 재사용 유틸 / 공유 코드 | `src/lib/` 또는 `src/shared/` |
| 테스트 | `tests/` (소스 옆에 흩뿌리지 않는다) |
| 사람이 읽는 문서 (스펙·룰·ADR) | `docs/` |
| AI가 남기는 기록 (로그·learnings) | `.dev/` |
| 에이전트 간 인계·조율 (인수인계서·작업보드·선점) | `.dev/collab/` (규약: `collab/README.md`, 여러 에이전트 협업 시에만) |
| **절대 수정 금지** | `out/`, `dist/`, 빌드 산출물, `*.lock` |

→ 둘 위치가 모호하면 **새 패턴을 발명하지 말고 먼저 물어본다.**

## 문서 분리: `docs/` vs `.dev/`

분리하지 않으면, 사람이 관리를 멈춘 순간 AI도 엉뚱한 맥락으로 일한다.

- **`docs/` = 사람의 책임 · 비즈니스의 진실** — 도메인 정의, 비즈니스 룰, API 스펙, ADR, 온보딩.
- **`.dev/` = AI의 작업 흔적** — troubleshooting 기록, 작업 로그, 실험 결과, 스크래치패드.
- AI는 작업 **전** `docs/`를 참고하고, 작업 **중/후** 배운 것을 `.dev/`에 남긴다.

## 에이전트 간 협업: `.dev/collab/` (여러 에이전트로 일할 때만)

여러 에이전트/세션이 한 프로젝트를 **이어서·동시에** 작업하면 조율 상태를 파일로 공유한다. 단일 에이전트면 만들지 않는다.

- **시작 전** `collab/handoff/`(이어받을 작업)와 `collab/BOARD.md`(진행 중)를 먼저 읽는다 — 같은 작업을 중복으로 잡지 않게.
- **선점**은 `collab/claims/<task>.md`로 표시하고 `BOARD.md`에 한 줄. **넘길 때**는 `collab/handoff/<topic>.md`에 인수인계서.
- **git 분리**: 인계서(`handoff/`)·결정(`DECISIONS.md`)은 **커밋**, 라이브 상태(`BOARD.md`)·락(`claims/`)은 **gitignore**(휘발성·머지 충돌 방지).
- 상세 프로토콜은 `.dev/collab/README.md`.

## 설정 스코프: User → Project → Folder (하위가 상위를 덮어쓴다)

범위가 좁을수록 우선순위가 높다. 하위는 상위를 **자동 상속**하되 **오버라이드**한다.

| 스코프 | 파일 | 무엇을 담나 |
|--------|------|-------------|
| USER | `~/.claude/CLAUDE.md` | 내 개인 습관·스타일 (모든 프로젝트 공통). git에 안 올림 |
| PROJECT | `./CLAUDE.md` | 이 프로젝트의 스택·컨벤션·경계 (팀 공유, **git 커밋**) |
| FOLDER | `./{폴더}/CLAUDE.md` | 그 폴더의 **특수 규칙** (그 폴더 작업 시에만 자동 로드) |

### 폴더 단위 CLAUDE.md 규칙 (중요)
- **모든 폴더에 둘 필요 없다.** 전역 규칙과 **다른 특수 제약**이 있는 폴더에만 둔다.
- 만드는 기준: 그 모듈이 프로젝트 전역 규칙으로 표현 안 되는 제약을 가질 때.
  - 예: `src/auth/` → "JWT만 사용, 외부 호출 금지" (템플릿: `src/auth/CLAUDE.md.example`)
- 둘 때도 **상속한 규칙은 다시 쓰지 말고**, 달라지는 부분만 오버라이드한다.

## `.claude/rules/` 작성 규칙

- 항상 적용할 규칙 → frontmatter `alwaysApply: true` (예: `code-style.md`).
- 상황별 규칙 → frontmatter `globs: [...]` 로 **조건부 로드** (예: `testing.md`, `security.md`).
- `CLAUDE.md`가 길어지면(>120줄) 주제별로 이 폴더로 쪼갠다 → 컨텍스트 절약.
- 한 파일 = 한 주제. 중복·대치되는 규칙이 생기면 주기적으로 점검해 정리한다(쌓이면 AI slop).

## 새 모듈/폴더 추가 절차
1. **이웃 폴더의 기존 패턴을 복제**한다 — 새 구조를 발명하지 않는다.
2. 위치가 애매하면 만들기 전에 물어본다.
3. 폴더에 전역과 다른 제약이 생기면 그때 `{폴더}/CLAUDE.md`를 추가한다(그 전엔 만들지 않는다).
4. 새 컨벤션이 생기면 `docs/conventions.md` 또는 적절한 `rules/`에 반영한다.
