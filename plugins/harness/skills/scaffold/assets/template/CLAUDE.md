## 폴더 지도 (어디에 무엇이 있는가)

| 폴더 | 용도 | 관리 주체 |
|------|------|-----------|
| `src/` | 비즈니스 로직 | 사람 + AI |
| `docs/` | 비즈니스 진실 · 아키텍처 · 스펙 (작업 **전** 참고) | **사람** |
| `tests/` | 검증 인프라 | 사람 + AI |
| `.dev/` | AI 작업 기록 · learnings · 디버깅 · 에이전트 간 인계(`collab/`) | AI |
| `.claude/` | AI 설정 (rules · skills · hooks) | 사람 |
| `out/` | 빌드 산출물 (**수정 금지**) | 자동 |

→ 새 파일은 이 구조를 따른다. "어디에 둘지" 모호하면 **만들기 전에 먼저 물어본다.**
→ 폴더 셋업/배치 상세 규칙: `.claude/rules/project-structure.md`

## 작업 규칙 (상세는 `.claude/rules/` 가 상황별로 자동 로드)

- 항상: 코드 스타일·네이밍 → `rules/code-style.md`

- 인증 / DB / SQL / `.env` 작업 시: `rules/security.md`

## 일하는 방식

- **"해줘"가 아니라 "같이 계획부터".** 모호하면 질문해 명확히 한 뒤 실행한다.
- **완료 기준을 먼저, 측정 가능하게 합의**한다(예: "테스트 통과 + 빌드 성공"). 충족할 때까지 반복.
- **만드는 관점과 검증하는 관점을 분리**한다. 코드뿐 아니라 화면·동작도 직접 확인.
- 같은 작업을 3번 반복하면 → Skill로. 같은 실수를 3번 하면 → 이 파일 또는 `rules/`에 규칙으로 남긴다.
- 여러 에이전트/세션이 이어서·동시에 작업하면 → `.dev/collab/`(인수인계서·작업보드·선점) 규약을 따른다. <!-- 단일 에이전트면 이 줄 삭제 -->

## 경계 (DON'Ts)

- `main`에 직접 push 금지 — 반드시 PR. `git push --force` 금지 (Hook으로도 차단 → 이중 안전장치).
- `out/`, 빌드 산출물, 잠금파일(`*.lock`) 직접 수정 금지.
- 비밀키 · `.env` 값 출력 / 커밋 금지. SQL은 **parameterized query만**.
- 삭제 · 배포 · 외부 발송 등 되돌리기 어려운 작업은 **실행 전 승인**받는다.
